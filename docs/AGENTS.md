# SopKit Codebase Documentation & SEO Rules

## Overview

[SopKit](https://sopkit.github.io) is a high-performance, developer-first utility ecosystem built with Next.js 16 (App Router) and Tailwind CSS v4. It provides **569+ free online tools** across `19` route-group categories (image, pdf, video, audio, text, developer, seo, youtube, generators, calculators, exam-tools, health, ai-tools, utilities, downloaders, company, content, blog, others) with a privacy-first, client-side sandbox processing model.

**Core Positioning**: "Privacy-first client-side tools — no data selling, no server uploads, 100% local sandbox." This is the unique value proposition against competitors (ipullif, smallpdf, ilovepdf, etc.) that upload user data to their servers.

## Core Architecture

### 1. Data Layer

- **tools.json** (`src/constants/tools.json`): Single source of truth for all 569+ tools. Contains id, name, description, route, category, extraSlugs (20+ per tool), popular flag, features, FAQs, howTo, article.
- **generated-manual-content.ts** (`src/data/generated-manual-content.ts`): Per-tool unique SEO content (whatItIs, features, howToUse, faqs, seoDescription). Generated by `scripts/generate-manual-content.cjs`. **This replaces all generic template content** — no `${name}` placeholder templates at runtime.
- **tools.ts** (`src/lib/tools.ts`): `getAllTools()`, `getToolByRoute()`, `getToolById()`, `getRelatedTools()`, `getCategoryBySlug()`.
- **config.ts** (`src/constants/config.ts`): Tool counts, URLs, social links.

### 3. SEO Infrastructure

- **generated-manual-content.ts**: Per-tool unique SEO content. **Every tool must have an entry here**. No fallback to generic templates (seoTemplates.tsx is deprecated).
- **seoTemplates.tsx** — **DEPRECATED**: Do not use. All content now comes from generated-manual-content.ts.
- **Sitemap** (`src/app/sitemap.ts`): Dynamic XML sitemap with priorities (popular=0.9, tools=0.75, SEO opportunities=0.84-0.9).
- **Robots.txt** (`src/app/robots.ts`): Allows 17+ crawlers including GPTBot, ClaudeBot, PerplexityBot. Disallows /api/, /admin/, /embed/.
- **StructuredData** (`src/components/shared/StructuredData.tsx`): JSON-LD per page (SoftwareApplication, FAQPage, HowTo, BreadcrumbList, Article).
- **LLM index** (`public/llms.txt`): AI-search-friendly tool index.

## SEO Rules — CRITICAL

### Content Rules (Anti-Spam, Genuine-Looking)
1. **Every tool gets unique content**: No two tools share identical text. Each tool's `whatItIs`, `features`, `howToUse`, and `faqs` are specifically about that tool's purpose.
2. **No generic template language**: No `${name}` placeholder replacements at runtime. All content is pre-generated per tool with its name, category, and purpose baked in.
3. **Write for humans, not crawlers**: Content must be genuinely helpful. Avoid keyword stuffing, repeated phrases, and "SEO-optimized" sounding paragraphs.
4. **Natural language variation**: Each FAQ uses different phrasing. Avoid starting every answer with the same phrase.
5. **Specificity over generality**: Instead of "This tool helps you process files," write "This tool converts JPEG images to PNG format" or whatever the specific action is.
6. **Privacy/security as brand identity**: The client-side sandbox positioning is a genuine feature, not a stuffed keyword. Every tool mentions it naturally in context.
7. **No spammy internal linking**: Links should be contextual and helpful, not "check out our other tools" lists stuffed with keywords.
8. **Realistic feature counts**: 5-8 features per tool, written as actual capabilities not keyword lists.
9. **FAQ answers must answer the question**: Each Q&A pair should be genuinely informative, not a rephrasing of the question with keywords inserted.
10. **HowTo steps must be actionable**: Each step describes a real browser action the user takes.

### SEO Technical Rules
1. **Title pattern**: `Free [Tool Name] Online — [Action] [Subject]`
2. **Meta description**: 150-160 chars, includes primary keyword, compelling CTA, mentions "client-side" or "private" naturally.
3. **H1 pattern**: `Free [Tool Name] — [Specific Action]` — unique per tool, descriptive, includes the primary keyword.
4. **ExtraSlugs**: 20+ per tool for long-tail SEO traffic. All redirect 301 to canonical route.
5. **Canonical URL**: Set explicitly on every page (the tool's primary route).
6. **OpenGraph + Twitter cards**: On every page via metadata export.
7. **Related tools**: Minimum 10 per tool page via `getRelatedTools()`.
8. **Schema types per page type**: Tool pages = SoftwareApplication + FAQPage + HowTo + BreadcrumbList + Article; Category hubs = CollectionPage; Homepage = WebSite + Organization + FAQPage.

### Tool Content Structure
Each tool in generated-manual-content.ts has:
- `whatItIs`: 600-800 word article. Section 1: what the tool does + how it works client-side. Section 2: "Why Choose [Tool]" — competitive positioning against server-side alternatives. Section 3: "Who Is It For" — target audience.
- `features`: 8 bullet points. Browser-local processing, zero data retention, offline capability, no signup, unlimited usage, fast speed, no watermarks/ads, free forever.
- `howToUse`: 5-step guide. Prepare -> Input -> Configure -> Execute/Preview -> Save. Each step references the specific file type/formats.
- `faqs`: 5 Q&A pairs. What does it do, Is it really free, Privacy protection, No signup required, Mobile compatibility.
- `seoDescription`: 1-2 sentence meta description.

## Anti-Spam Quality Checks
Before deploying any SEO content, verify:
- [ ] No two tools have identical feature lists or FAQ answers
- [ ] Content sounds like a real person wrote it (varie sentence length, use contractions, avoid robotic lists)
- [ ] Keywords appear naturally — no stuffing 5+ keywords into one sentence
- [ ] Each tool's page is genuinely useful (the tool actually does what the content claims)
- [ ] No excessive bolding of every keyword phrase
- [ ] FAQ answers vary in structure between tools in the same category
- [ ] The "client-side sandbox" message appears once in article and once in FAQ, not repeated in every section
- [ ] H1 tags are unique across the entire site (no two tools share the same H1)

## Developer Workflow

### Adding a New Tool
1. Append entry to `src/constants/tools.json` with id, name, description, route, category, extraSlugs, popular flag.
2. Run `node scripts/generate-manual-content.cjs` to regenerate `src/data/generated-manual-content.ts`.
3. Create folder and `page.tsx` in `src/app/(category)/tool-slug/` — export metadata object, use ToolLayout wrapper.
4. Map component in `IntentToolDispatcher.tsx` (if the tool has interactive UI).
5. Verify: sitemap includes it, robots.txt doesn't block it, LLM index updated, related tools include it.

### Regenerating SEO Content
```bash
node scripts/generate-manual-content.cjs
```
This reads all tools from tools.json and regenerates unique per-tool content in generated-manual-content.ts.
